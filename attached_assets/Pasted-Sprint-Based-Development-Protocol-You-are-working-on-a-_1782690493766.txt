Sprint-Based Development Protocol

You are working on a mature codebase that has already completed multiple stabilization phases.

The following engines are considered stable, verified, committed, and pushed to GitHub:

- Scanner Engine
- Feature Engine
- Diff Engine
- Dependency Engine
- Impact Engine
- Regression Engine
- Root Cause Engine

Treat these components as the project's trusted baseline.

Critical Operating Rules

- DO NOT re-read the entire project.
- DO NOT perform another full codebase audit.
- DO NOT review previously completed engines.
- DO NOT redesign previously completed architecture.
- DO NOT refactor outside the current task.
- DO NOT modify unrelated files.
- DO NOT consume tokens analyzing completed work.

Previously completed work should only be revisited if:

- a failing test proves a dependency issue,
- a compile/runtime error requires investigation,
- or the current task cannot be completed without a minimal dependency change.

Assume the current baseline is correct unless objective evidence proves otherwise.

---

Development Workflow

Follow this workflow exactly.

1. Read ONLY the files required for the current task.
2. Run ONLY the tests related to the current task.
3. Write comprehensive edge-case tests.
4. Run those edge-case tests.
5. Fix confirmed bugs only.
6. Re-run the task-specific tests.
7. Run the COMPLETE test suite ONCE and ONLY after all fixes are finished.
8. If every test passes:
   - git add .
   - git commit
   - git push origin main
9. Produce a concise engineering report.
10. STOP immediately.

Never continue into another task.

Never start another engine.
